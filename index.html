Business Case

HM Revenue & Customs (HMRC) intiated a programme to replace its existing customs system, CHIEF (Customs Handling of Import and Export Freight), with a new Customs Declaration Service (CDS). HMRC is changing its customs system to comply with legislative requirements that were established before the UK voted to leave the European Union (EU). The new system will be an integral part of the UK’s overall customs regime. The UK collects around £34 billion annually from customs and excise duties and value added tax (VAT) on transactions at the border. In 2017, around £820 billion of goods crossed the border. It is vital to the UK’s economy and consumers that trade continues to operate smoothly across the border, for example by maintaining the trade flow of perishables such as food products.
The UK’s decision to leave the EU and the customs union has important implications for the CDS programme. Depending on the outcome of EU exit negotiations, from March 2019 at least 145,000 traders who currently import or export goods solely within the EU might be required to make customs declarations for the first time; and the number of customs declarations could increase from around 55 million to around 255 million each year.

People Source Consulting have been engaged by HMRC to deliver the Customs Declaration Service (CDS) integration layer for HMRC Enterprise Integration Services (EIS). The HMRC Chief Digital and Information Office (CDIO) and EIS are responsible for all of HMRCs core Integration services that deal with over a billion internal and external transactions every year; connecting 100s of HMRC services, developed in a wide variety of technologies.
EIS provides a fundamental framework for delivering Application Programming Interface (API) access to HMRC core systems and enabling their digital agenda, API strategy and transformation roadmaps for all their services. They are utilising modern open source products and working in agile methodologies and approaches to deliver flexible costeffective integration solutions.

CDS is a key transformation programme and all of the key integration with other HMG systems will be carried out as part of this delivery. The following Projects are in scope for delivery:

    EIS Architecture Overview

    Summary

    EIS is part of Enterprise Platform Services, cross-cutting horizontal organizations and responsible for building and supporting Secure Integration between HMRC's internal systems and external entities, including other Govenment Departments. EIS is also responsible for on-boarding and managing relationships with 3rd parties (including commercial SW), utilising HMRC's external facing APIs for integration. EIS enables the collection of all revenue and customs income. Hundreds of billions of pounds worth of transactions traverse integration services per year.

    The logical components within the system are:

  • Multi-Channel Digital Tax Platform (MDTP): A digital platform with a common infrastructure that links existing and new systems to provide consistent and responsive services.
  • Perimeter Proxy + WAF (Nginx): BIG IP Firewall system. The Web Application Firewall (or WAF) filters, monitors, and controls HTTP traffic to and from the MDTP. This system maps the service endpoints between the Front-end client system and the CDS.
  • API Gateway (WSo2 APIM): This is the internal firewall system in CDS. This authenticates and authorises all the RESTful requests based on OAUTH tokens.
  • API Key Manager: Stores the OAUTH token keys of all the MDG services. This information is used by the MDTP to access the API services.
  • Internal Proxy (Nginx): This is the load balancer of the Enterprise Service Bus (ESB). This sytem manages the worker nodes.
  • Enterprise Service Bus (ESB): This is a WSO2 engine that enables communication between the front-end and back-end services through mediation, transformation and messaging. All the the API/ Proxy services are deployed here to facilitate the communication.
  • AV Service: Provides Anti-virus service for the API Gateway.
  • Transform and Validate Service: This system validates Front-end system requests with respect to the Front-end schema, transforms the requests and then validates it with respect to the Back-end schema. Likewise, it will also validate-transform-validate the back-end response.
  • JMS Service: A point-to-point model that maintains queues of incoming messages for interaction with queue based Back-end system.
  • AMQP Service (RabbitMQ): The system routes, buffers and persist the messages according to the specified rules. Used when the Back-end is a RabbitMQ based system.
  • File Transfer Subnet: File transfer system containing file transfer agents and managers to facilitate automated processing and transfer of files from the Application Buckets and MDG CDS Buckets.

    • Deployment Process

    Single release branches are deployed to environments controlled by Jenkins. The release branch only contains the changes for the specific feature allowing multiple releases in a day to support business needs. The release branch will progress through the environment and have been deployed multiple times before deployment in production. The delpoyment process is as follows:

  • Developers commit the code to Apache Subversion.
  • Building and deploying in Dev and CI environments: BOM code submission in SVN triggers autoseed job on Jenkins. The autoseed job creates the pipeline.
  • The executable artifacts built by the pipeline jobs is pushed to NEXUS on FAST-P. Also the pipeline jobs deploy the executable on the corresponding environments such as DEV, CI or NFR.
  • The executable artefacts built by the pipeline jobs are also be saved in Artifactory.
  • After a successful execution of a pipeline and if the journey is ready to be deployed to higher environments a release Jenkins job is executed to generate a release branch in GitLab. This release branch is used for deployment in higher environments.
  • GITLab and Artifactory are provisioned and maintained in CDG's ALM VPC. The deployment artefacts and configuration for non-production and production environments are picked up from Artefactory and GitLab respectively. When the Ansible is executed it picks the executable from Artifactory and deploys in relevant environment.
  • The Ansible configuration in GitLab that gets executed during journey deployments will pickup executable artefacts from Artefactory for productions and all integration test environments.
  • EIS are required to follow a change request process. Also the backend endpoints are provided by CDOS team via Ansible programme_vars.
  • To test EIS intefaces in DEV, CI and NFR where a backend is not available, a backend mock server that holds the backend mock responses is used in End-to-End testing.
  • Kafka and the ELK stack are used for centralised log management of application logs and generated journey stream events. Sensu is used to monitor system metrics and service availability.
  • Nginx Plus provides WAF capability and routing for external requests originating from Digital frontend applications incoming in to prod and non-prod VPCs. Network connectivity and access is managed using a combination of host based firewalls (IP Tables) and AWS security groups.
  • Access to APIs exposed externally is provided via the gateway component of WSO2 API Manager. WSO2 API gateway requires OAUTH tokens for access.
  • DevOps team develop Terraform for infrastructure provisioning.
  • Ansible is used for configuration management. GITLab on CDG ALM VPC is used as a code repository.

    Integration Patterns

    1. External Digital to Internal Systems Messaging Pattern

    EIS follows a specific pattern when implementing messaging patterns for external digital based systems. Generally the externatal system will be a web based digital aplication which will call an EIS public API interface. The process will follow the following steps:

  • External web based application will send an JSON/XML based HTTP request.
  • The request will be channelled through the WS02 API gateway - which will perform the required Authentication and Authorisation checks as well as route the request to the appropriate back-end service/s.
  • The next phase will be the external facing Data Tier, which houses a message adapter that will accept the message and invoke the required service/s.
  • Nginx will provide load balancing capabilities and WAF security.
  • The Corporate Tier will house the Assets Microservices that will perform the necessary validations and request transformation.
  • Message will be sent to the back-end HoD systems.

    • 2. Message Queue Pattern

    EIS implements different Queueing technologies like Rabbit MQ, IBM MQ and JMS, depending on the specific requirements of the journey. Queues are required in order to implement message persistence or guaranteed messages delivery.
    When a message has been published to a queue, the receiving client will subscribe to the queue in a pre-determioned schedule in order to poll for the specific request payload. The assets microservice will perform the usual schema avalidations, etc. prior to posting the message to the queue.
    Generally a queue based message pattern is Asynchronous and will be required to ensure that the message is received by the end receiving system. Therefore, if the receiving system is not available, there will be a pre-configured number of retries over a set period of time to ensure message persistence.

    3. Back-Front Systems Message Pattern

    EIS will also cater to messaging from internal Head of Duty (HoD) systems to external systems. Thes epatterns are basically the reverse of pattern 1, and will follow similar architecture and protocols. The key things to consider are;

  • Validation and transformation of messages via in-built microservices that reside in the assets repository.
  • Ensuring that the message reaches the right end-clients via secure end-points.
  • HTTPS connectivity with external clients, to ensure that the transport of information is done n a secure manner.

    • 4. Internal Systems Messaging Pattern

    This pattern caters to the process of routing and managing messages internally. EIS manages the internal messaging between different systems in the HMRC estate. This includes various HMRC systems as well as systems belonging to Other Government Departments (OGDs). These messages will not be required to be routed through an API gateway, as the components are within the secured zone (corporate tier).
    Generally most OGDs follow the same Architectural principles and Protocols when it comes to messaging (i.e. SOAP), therefore the transformation of messages are rarely necessary. However, some file transer patterns may require adherence to uniform formats.

    5. File Transfer Patterns

    EIS implements several patterns that cater to file transfer between external-interal, internal-external and internal-internal systems. These journeys pre-dominantly encompass legacy systems and protocols that have been established for a long time. Therefore, any file transfer jurneys that arr required to transfer either from or to these legacy systems will be required to adhere to these protocols. The following specific patterns are implemented:

    AWS External Party to AWS EIS using JobScheduler

    Process Steps

  • An HTTPS 'Ready for Transport' call will be received from the source system, with which a JobScheduler Order will be created. This will in turn instantiate the appropriate JobScheduler Job Instance. All external synchronous HTTPS requests will be routed through an AWS hosted WAF to the correct end-point by an API GW
  • The JS Job Instance will pull the file from an external location using FTPS / HTTPS into the EIS DT Virus Scanning server's local file system.
  • A Virus Scan will ve excuted on the inbound file, and if a virus has been detected, the inbound fie will be deleted from the local file system and the journey will be terminated as an Error.
  • If a virus is not detected, the file will be moved to the external facing S3 bucket /from folder (clean area) for this specific journey.
  • The JS Agent in the AWS EIS CT tier will connect to the external facing EIS S3 Bucket and pull the file into the internal EIS S3 Bucket.
  • The JS Agent in the AWS EIS CT tier will then make an XML over HTTPS service call to the destination system/application informing them that the file is ready for consumption. The job instance will then terminate successfully.
  • The destination system/application may also poll it's S3 Bucket folders in order to detect when a file is ready for processing. In this scenario, the JS will not send any notification.
  • The destination system/application will consume the file and then remove it from the EIS internal S3 bucket.
  • When terminating an instance, the JS will tidy up any files generated during the process, leaving only a copy of the file in the EIS Internal S3 Bucket. This is required as a backup in case of a failure in the file transfer process.
  • A JS Housekeeping job will be run after a configured amount of time, to archive and delete the internal S3 bucket files. Ideally the destination system/application must remove the file from the internal S3 bucket.

    • Alternative Steps

  • If the file is received from a trusted source (usually an internal source), then the virus scanning step may not be required.
  • Additional steps to Encode, Decode, Encrypt, Decrypt, etc. can be added to the JS jobchain.
  • If the file transfer occurs between two legacy systems and no AWS S3 Buckets are present, the the transfer pattern will be limited to legacy system servers like FTP/SFTP/FTPS/UTM/etc.

    • Metadata File

    It is expected that a Metadata file will always be present with each file that needs transferring. A Metadata file is Mandatory in all applicable file transfer patterns. If the source system/application does not provide a Metadata file, EIS will generate it's own.

    Platform Migration

    Summary

    A decision was made to migrate the existing HMRC EIS Systems and Technology from it's old monolithic form to a more modern architecture.The objective was for EIS to safely exposes HMRC's organisational capabilities as highly cohesive, loosely coupled services using securely orchestrated microservices. Key challenges are to implement API and File Transfer solutions that work alongside and in time, replace existing monolithic systems and processes. This will be achieved by imeplmenting in-house built Microservices based technology.

    Integration Framework

Each with many environments, different maturity, different tooling and supported by different teams. This meant that frameworks were tightly coupled to specific projects.
